## Research Question 1: MDX Component Integration in Docusaurus v3

**Decision**: Register `<RagChatbot>` as a global MDX component via `docusaurus.config.js`
rather than importing it per-file.

**Rationale**: Docusaurus v3 supports `mdxComponents` in the theme config. A global registration
eliminates a repeated `import RagChatbot from '@site/src/components/RagChatbot'` line at the top
of every chapter file, reducing boilerplate and ensuring consistency. If the import is missing
from even one file, the build fails silently on that page's widget render.

**Alternatives considered**:
- Per-file import: simpler for one-off use but error-prone at scale (4+ files, 16 chapters total).
- MDX remark plugin to auto-inject: over-engineered for this scope.

**Resolution**: Global registration via `docusaurus.config.js` theme config. Each MDX file uses
`<RagChatbot context="module-1" />` directly without an import statement.

## Research Question 2: Docusaurus Sidebar Format (v3)

**Decision**: Manual sidebar definition in `sidebars.js` using the `category` object type.

**Rationale**: Autogenerated sidebars derive order from filesystem sort, which is alphabetical
by default. Manual definition guarantees chapter 1 â†’ 2 â†’ 3 â†’ 4 ordering regardless of filename
or frontmatter changes.

**Alternatives considered**:
- `autogenerated` sidebar: loses ordering control; requires `_category_.json` per directory.
- `position` frontmatter in each MDX file: works but is fragile (renumbering requires touching
  every file).

**Resolution**: Manual `sidebars.js` category entry with an explicit `items` array. Docusaurus
doc IDs are derived from file paths relative to `docs/` (e.g., `module-1/chapter-1-middleware`).

## Research Question 3: RAG Chunking Boundary Strategy

**Decision**: Use `##` (H2) headings as the primary chunk boundary for the Qdrant ingestion
pipeline. H3 headings are sub-chunks within the same semantic block and MUST remain under
their parent H2's context.

**Rationale**: Standard markdown text splitters (LangChain `MarkdownHeaderTextSplitter`,
LlamaIndex `MarkdownNodeParser`) split on H2 by default. Our constitution's FR-004 requires
headings to be self-describing so chunk retrieval is accurate. H1 is the document title and
is not a split point.

**Alternatives considered**:
- Sentence-level chunking: loses structural context; RAG answers lose section framing.
- H3-level chunking: too fine-grained; creates hundreds of tiny chunks per module with high
  semantic overlap.

**Resolution**: Each chapter MUST have a minimum of 4 H2 sections. H2 text MUST include the
chapter topic as a qualifier (e.g., `## The Hardware Abstraction Problem in Robotics`, not
`## The Problem`). This produces â‰Ą3 retrievable content chunks per chapter (excluding the
Learning Objectives and Summary sections which serve as metadata chunks).
