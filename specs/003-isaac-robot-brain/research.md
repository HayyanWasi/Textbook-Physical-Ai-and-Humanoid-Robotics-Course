## Research Question 1: RagChatbot Import Strategy in Existing Chapters

**Decision**: Per-file import of `RagChatbot` from `@site/src/components/RagChatbot`.

**Rationale**: Inspection of existing Module 1 and Module 2 MDX files confirms the established
pattern: every chapter opens with `import RagChatbot from '@site/src/components/RagChatbot';`.
Global MDX component registration was considered in the Module 1 research phase but was not
implemented — the component is not listed in `docusaurus.config.ts` under `mdxComponents`.
Deviating from the established pattern would introduce inconsistency without benefit.

**Alternatives considered**:
- Global registration via `docusaurus.config.ts` `mdxComponents`: cleaner, but would require
  touching `docusaurus.config.ts` and risks breaking existing chapters if misconfigured.
- MDX remark plugin to auto-inject the import: over-engineered for 4 files.

**Resolution**: Per-file import. Every Module 3 chapter opens with the same import line used by
Modules 1 and 2. No changes to `docusaurus.config.ts` required.

## Research Question 2: Sidebar File Format (TypeScript vs JavaScript)

**Decision**: Extend `frontend/sidebars.ts` (TypeScript) by appending a third category object
to the existing `tutorialSidebar` array.

**Rationale**: The Module 1 plan documented `sidebars.js` (JavaScript), but the live file is
`frontend/sidebars.ts` with a `SidebarsConfig` TypeScript type annotation. The file must remain
TypeScript-typed. The Module 3 category is appended as a third entry — after Module 1 and
Module 2 — using the same structure pattern.

**Alternatives considered**:

- Using `_category_.json` + autogenerated sidebar: loses explicit chapter ordering.

**Resolution**: Manual TypeScript category entry appended to `tutorialSidebar` in
`frontend/sidebars.ts`. Docusaurus doc IDs use the `module-3/chapter-N-slug` format (no
extension), matching the pattern established by Modules 1 and 2.

## Research Question 4: Gazebo vs. Isaac Sim Disambiguation Placement

**Decision**: Chapter 10 MUST include `## Gazebo vs. Isaac Sim: A Direct Comparison` as a
top-level H2 section — not nested inside another section — so it forms an independent RAG chunk.

**Rationale**: FR-003 mandates an explicit comparison section. SC-005 requires the knowledge
retrieval system to return semantically distinct results for the "Gazebo" vs. "Isaac Sim" query
pair. If the comparison is embedded inside a larger section (e.g., under `## Two Simulators`),
the RAG chunker will bundle it with surrounding content and blur the semantic boundary.

**Alternatives considered**:
- Inline comparison table inside a broader section: easier to write but creates RAG chunk
  contamination — a query for "Gazebo" retrieves Isaac Sim content and vice versa.
- Separate document for the comparison: unnecessary fragmentation for four-chapter scope.

**Resolution**: Standalone H2 section `## Gazebo vs. Isaac Sim: A Direct Comparison` in
Chapter 10, positioned after the individual descriptions of each simulator.

## Research Question 5: VSLAM Abstraction Boundary (Chapter 11)

**Decision**: Chapter 11 uses the "mental map + landmark photograph" analogy as its primary
frame. Prohibited terms: "matrix," "Jacobian," "covariance," "eigenvalue," "Kalman," "graph
optimization," "factor graph," "bundle adjustment."

**Rationale**: SC-003 requires that a student with no GPU or CUDA background can read all
chapters without encountering unfamiliar technical jargon. The VSLAM literature is saturated
with linear algebra terminology that would immediately exclude non-mathematical readers.
The "navigating an unfamiliar city by photographing landmarks" analogy maps cleanly to:
- Feature extraction → noticing distinctive buildings
- Feature matching → recognising the same building from a different angle
- Pose estimation → inferring your location from known landmarks
- Loop closure → realising you've returned to a street you photographed earlier

**Alternatives considered**:
- Using simplified equation notation with plain-English captions: still introduces symbols
  that trigger anxiety in non-mathematical readers; violates FR-004 spirit.
- Referencing external VSLAM papers with "for deeper reading" links: acceptable for optional
  enrichment but not as a substitute for the core conceptual explanation.

**Resolution**: Zero mathematical notation. All VSLAM concepts explained exclusively through
the "city navigation" analogy and reinforced with labeled diagrams (described in prose, as
the MDX files are text-only until image assets are added in a later iteration).

## Research Question 6: Bipedal vs. Wheeled Navigation Differentiation (Chapter 12)

**Decision**: Chapter 12 explicitly frames bipedal navigation challenges as three distinct
constraints absent from wheeled navigation: (1) center-of-mass stability, (2) discrete
footstep placement on valid terrain, and (3) gait cycle timing.

**Rationale**: FR-005 mandates that Chapter 12 distinguishes bipedal from wheeled path
planning. SC-006 requires students to trace the full pipeline from camera to footstep decision.
Without a crisp enumeration of the structural differences, students may assume Nav2 works
identically for humanoids and differential-drive robots.

**Alternatives considered**:
- Describing Nav2 generically and noting humanoid adaptations as "extensions": undersells the
  architectural differences and fails FR-005's "distinct requirements" language.
- Including footstep planner pseudocode: violates the no-algorithm-derivations constraint.

**Resolution**: Three-constraint frame introduced early in Chapter 12, then each constraint
is addressed by name in subsequent sections. No pseudocode. Nav2's role as the planning
orchestrator (not the balance controller) is clarified to avoid scope confusion.
